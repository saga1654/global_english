LiveEnglishClass 모바일 퍼블리싱 관리 메인 (HTML/CSS 가이드)

개요

본 가이드는 "LiveEnglishClass" 모바일 퍼블리싱 작업 시 업무공유를 위해 제작되었으며 DTD, Encodng 방식을 비롯 HTML, CSS의 작성 규칙 및 네이밍 규칙, 접근성 확인 사항 등에 관한 내용을 담고 있다.

  • LiveEnglishClass는 W3C에서 권고한 웹표준과 국가표준인 지침에 맞춰 웹/모바일 접근성을 준수하여 제작한다.
  • LiveEnglishClass는 internet explorer 7.0 버전 이상, Chrome, Safari, Firefox, Opera 최신 버전 브라우저에 맞춰 제작한다.

DTD (Document Type Definition)

LiveEnglishClass 모바일 페이지의 DTD는 HTML5를 사용하며 아래와 같이 선언한다.

<!DOCTYPE html>
            <html lang="ko">
            ...
            </html>
1. HTML5 특징
  • 기존 HTML, XHTML에 비해 DTD 선언이 매우 간단하다.
  • section, article, nav 등 구조를 나타낼 수 있는 tag들이 다양하게 생성되어 전 HTML, XHTML에 비해 의미있는 마크업이 가능해졌다.
  • 다양한 폼 type이 새로 추가되어 더 쉽고 강력해졌다.
  • 외부 플러그인없이 VIDEO, AUDIO 태그와 같은 기본 태그로 멀티미디어 요소가 지원된다.
2. HTML5 지원 현황

    현재 ie8부터 일부 기능을 지원하고 있으며, html5를 지원하지 않는 브라우저를 위해 가벼운 js를 이용해서 해결한다.

3. HTML5의 새로운 요소
  1. 1) section

    2. 일반적인 문서 또는 프로그램 섹션을 나타내며 보통 제목으로 시작하는 컨텐츠의 의미적 그룹을 나타낸다. 요소의 내용을 배포해도 이치에 맞다면 section 요소 대신 article요소를 사용하는 것이 맞으며 요소에 디자인적인 스타일이나 스크립트의 편의성을 위한 목적으로만 wrapper를 만든다면 div를 사용해야 한다. 섹션요소를 사용하기에 따라 제목 레벨을 고민하지 않고 h1요소를 사용해도 된다.

  2. 2) nav

    3. nav요소는 다른 페이지 또는 동일 페이지의 다른 부분으로 이어주는 네비게이션 링크로 구성된 섹션을 나타내며, 주된 Navigation Block을 형성하는 섹션이 nav요소에 적합하다. 페이지의 '이전','다음'과 같은 페이지의 페이징 부분의 섹션을 담을 때 사용할 수도 있다.

  3. 3) article

    4. article요소는 문서나 페이지, 사이트에서 독립적으로 배포 또는 재사용할 수 있는 섹션을 나타낸다. article요소는 중첩되어 사용될 수 있으며, 중첩되어 사용되었다면 내부의 article요소는 외부의 article요소와 관련이 있어야 한다. 예를 들면 blog 내용이 article요소안에 있고 댓글영역이 article로 되어있는 경우를 들 수 있다.

  4. 4) aside

    5. aside 요소는 주위 내용과 접점을 이루면서 주위 내용과는 분리된 것으로 간주된다. aside요소가 나타내는 섹션은 출력시 사이드바 형태로 표현되는 것이 일반적이다.

  5. 5) hgroup

    6. hgroup요소는 섹션의 제목을 나타내는 요소인데 섹션의 제목이 몇 개의 레벨을 가질 때, 예를 들어 부제목이나 제목 요소들을 그룹 짓기 위해 사용한다.

  6. 6) header

    7. 일반적으로 header요소는 보통 섹션의 제목(h1~h6, hgroup)을 포함하지만 섹션의 차례나 검색 폼, 관련 로고 등을 감싸는 용도로 사용할 수 있다.

  7. 7) footer

    8. footer 요소는 가장 가까운 조상 요소인 섹션 컨텐츠 또는 섹션 루트의 푸터를 나타낸다. 푸터는 일반적으로 작성자가 누구인지, 연관된 문서에 대한 링크, 저작권 정보, 기타 흡사한 정보들을 담고 있다. footer는 일반적으로 섹션의 "마지막"에 나타나지만 꼭 그렇진 않다.

  8. 8) address

    9. address요소는 가장 가까운 조상 요소인 article 또는 body 요소의 연락처 정보를 나타낸다. address요소에 임의의 주소(예를 들어 우편주소)를 사용하면 안되는데 그 주소가 실제 연락처라면 예외로 한다. 우편 주소를 마크업할때 적합한 요소는 p 요소이다. address요소는 연락처 정보 이외의 것을 담고 있으면 안되며 보통 다른 정보와 함께 footer요소안에 넣는다.

  9. 9) figure

    10. 이미지나 비디오, 일러스트, 다이어그램, 일반 텍스트 등을 하나의 단위로 묶어 줄때 즉, 텍스트건 다른 요소건 태그들을 하나의 블록으로 묶고 캡션을 포함할 때 유용하게 사용할 수 있다. <figcaption>태그는 <figure>로 묶인 박스를 설명하는 캡션으로 사용한다.

  10. 10) time

    11. 시간이나 날짜를 나타내주는 텍스트를 시스템이나 웹브라우저가 시간이나 날짜로 인지시키는 요소이며 datetime 속성을 가진다.

  11. 11) form의 type 추가

    12. datetime, week, month, time, email, url, color, search, tel, file(multiple, accept)과 같은 type이 새로 추가 되었으며 모바일페이지 마크업쪽에서 사용하기로 한다.

Encoding

1. HTML

HTML문서는 반드시 인코딩 정보를 선언한다. 신규 HTML문서를 작성할 때 기본 인코딩은 UTF-8을 원칙으로 한다. 

<meta charset=”utf-8”>

※ UTF-8 : 한글은 물론 세계의 다양한 언어를 문제 없이 표현할 수 있는 장점이 있는 반면 한글 한 자를 3bytes로 처리하여 문서 크기가 다소 커지지만, 공백이나 영문 한자는 1byte로 처리하는 유연한 방식이다.

2. CSS

CSS 도 역시 UTF-8을 원칙으로 하며 CSS 저장 시 UTF-8로 저장하며, 파일 최상단에 아래 코드를 삽입한다.

@charset "utf-8";

문서 제목

접근성을 고려하여 간결하고 unique한 문서 제목을 사용한다.

                    <title>로그인- LiveEnglishClass</title>
                <title>회원가입 - LiveEnglishClass</title>
                <title>화면 title - site 명</title>

HTML 규칙

1. 코드 작성 규칙

1) 기본 작성 규칙

  • HTML은 위의 DTD의 명세에 맞게 작성하며, 개발 이관 전 W3C Validation Check를 통과해야함
  • 모든 요소와 속성은 소문자로 작성함
  • 속성값은 큰따옴표("")로 묶음
  • 특수기호는 character entity reference 사용 함 ref) HTML Entity 참조

2) 들여쓰기/빈줄 규칙

  • · 부모 구조 관계를 나타내며 코드 가독성을 위해 들여쓰기를 준수하고 들여쓰기 기준은 space 4칸으로 함
  • · 의미있는 객체를 구분하기 위해 코드 그룹 간 1줄씩 빈 줄을 만드는 것은 허용하며 빈 줄의 간격은 1줄을 초과하지 않음

3) 주석

                주석 시작 : <!-- 주석내용 -->
            주석 종료 : <!-- //주석내용 -->
            퍼블리셔 전달 : <!-- [P] 별도 class 추가 -->
            개발자 전달 : <!-- [D] 개발시 수정가능 -->
  • 일반 컨텐츠 블록 구분을 위한 주석과 퍼블리셔, 개발자에게 전달용 주석([P],[D])을 구분해서 사용함
  • 시작과 종료 주석의 내용은 동일해야 하며 주석기호와 주석 내용 사이에는 공백 한 칸이 있어야 함
  • 컨텐츠 블룩 구분이외의 퍼블리셔, 개발자 전달 용으로 작성한 주석은 실제 서비스 적용 전 삭제함

4) 선언순서

코드간 일관성을 위해 아래와 같은 순서대로 작성한다.
type - id - name - alt- title - class - style

2. 요소 작성 규칙
  1. 1) div
  2. 기존 버전의 html과 달리 html5에서는 디자인요소 적용을 위한 그룹핑이나 스크립트때문에 필요한 경우를 제외하고는 div사용을 지양한다.
  3. 2) section
  4. 의미적으로 제목을 갖는 컨텐츠의 의미적 그룹을 나타내며 section 안에서 제목은 레벨을 고민하지 않고 h1으로 마크업한다.
  5. 3) article
  6. 문서나 페이지, 사이트에서 독립적으로 배포 또는 재사용할 수 있는 섹션을 나타내며 중첩 사용가능하다.
  7. 4) header
  8. 컨텐츠의 머리글 역학을 하며 보통 컨텐츠의 제목은 물론, 차례, 검색폼, 관련 로고 등을 감싸는 등의 역할을 한다.
  9. 5) footer
  10. footer요소는 가장 가까운 조상 요소인 섹션 컨텐츠 또는 섹션 루트의 푸터를 나타낸다.
  11. 6) form, fieldset, legend
  12. form, fieldset, legend 요소는 함께 사용하며 필요에 따라 개별적으로 사용할 수 있다.
  13. 7) input 요소
  14. HTML5에 새로 등장한 type(ex:search, url, tel 등)은 사용하지 않으며 label과 연결시키며 연결할 label이 존재하지 않는 경우 input 요소에 attribute 선언이 필요한 경우 아래를 참조한다.
    • input checked="checked"
    • input disabled="disabled"
    • input required="required"
  15. 8) textarea
  16. textarea는 cols, rows Attribute는 페이지에 맞게 임의로 제공하며 최소 30,5 이상이 되도록 선언한다. 연결할 label이 존재하지 않는 경우 title을 사용하여도 된다.
  17. 9) 각 버튼성 요소
  18. 단순 링크성 버튼은 a 태그를 이용하고, 폼 전송 역할은 button type="submit" 을 이용하며 UI조작에 필요한 기능성 버튼은 button type="button"을 이용한다.
  19. 10) 표 요소
  20. 레이아웃용 table은 지양하고 데이터테이블과 디자인이 table식으로 구성 된 fieldset 형식의 요소에만 사용한다.
    • HTML5에서는 summary 속성은 사용하지 않으며 caption 이나 table밖에 요약성 정보를 필수로 제공한다.
    • 제목성 컬럼 th에는 scope 속성을 반드시 선언하며 셀병합 테이블의 경우는 scope대신 id, headers 속성을 사용한다.
  21. 11) iframe
  22. iframe은 페이지 성능에 영향을 주기 때문에 가급적 사용하지 않는다. 부득이하게 사용할 경우 title에 제목을 반드시 표기한다. 내용이 없는 빈 프레임인경우 title="빈프레임" 이라고 명시해준다.

CSS 규칙

1. 코드 작성 규칙

1) 기본 작성 규칙

  • CSS는 사용 가능한 hack과 css3 속성을 제외하고 W3C Validation을 통과하여야 한다.
  • 모든 속성은 영문 소문자로만 작성한다.
  • 공백이 포함된 폰트명, 한글 폰트명, 문자열 데이터 타입, filter 속성의 파라미터값은 작은따옴표('')를 사용한다.
  • 생략해도 무방한 작은따옴표('')는 생략한다. ex) background:url(image.png);
  • 축약이 가능한 속성은 가능한 축약해서 사용한다.
  • 마지막 속성의 세미콜론은 지양한다.
  • 쉼표로 구분되는 선택자 간 공백은 제거한다.
  • 속성 간 공백 및 속성 값 사이 공백은 제거한다.
  • 속성, 속성 값 사이 줄 바꿈은 허용하지 않는다. 단, media query 처럼 중괄호가 중첩되는 경우는 예외로 함
  • 특수한 경우를 제외한 일반적인 코딩 시 스타일 속성의 inline 사용을 지양함
  • 다른 곳과의 충돌을 고려해 공통요소가 아니라면 unique한 클래스를 사용함

2) 속성 선언 순서

속성을 선언할 때는 레이아웃과 관련이 큰 것부터 시작해서 레이아웃이 무관한 것 순서로 선언한다.
display, overflow, float, position, width, height, margin, height, margin, padding, border, background, font ..

3) 폰트 선언

영문폰트만 선언할 경우 특정 브라우저에서 폰트를 올바르게 출력하지 못하는 경우가 있으므로 한글 폰트 선언 시 한글, 영문 폰트를 모두 선언한다.

4) 주석

/* 주석 */
  • 주석 기호와 주석 내용 사이에는 반드시 공백 한 칸이 있어야 주석 기호와 주석 내용 사이의 줄바꿈은 허용하지 않는다.
  • 종료 주석은 사용하지 않는다.
  • 작업간 편의를 위해 작성한 주석은 서비스 적용 시 반드시 삭제한다.

5) File Verification

CSS 파일 작성 후 NHN에서 제공하는 N-MET(NHN Markup Enhancement Tool)을 사용하여 공백, 마지막 ; 삭제, 속성 선언순서 등을 최적화한다. 단, css3속성은 변환 시 bug가 있는 경우가 있으니 별도로 확인해야한다.

2. 선택자 사용 규칙

  • 공통 선택자는 성능에 영향을 미치므로 사용지양한다.
  • id선택자는 개발쪽과 충돌이 날 수 있으니 퍼블리싱영역에서는 한 문서에 반드시 한 번만 사용하며 앵커로서 가치가 있는 경우만 id를 사용한다.
  • 크로스브라우징을 이용해 가상클래스를 사용한다.(ex: :first-child)
  • 크로스브라우징을 이용해 복합선택자(하위선택자, 자식선택자, 인접선택자)를 사용한다. (ex: A B, A>B, A+B)

네이밍 규칙

1. HTML 파일명

LiveEnglsihClass의 모바일 대상을 크게 메인페이, 메인페이지 광고, 로그인 및 회원가입 페이지, 마이페이지, 서브페이, 수강신청, 레벨테스트 총7개로 나눠 이니셜로 구분한 후 각 타입별 1depth메뉴는 영문화하여 표기하고 2depth부터는 숫자화하여 표기한다. 자세한 규칙은 아래를 참조한다.

2. ID명

아이디명은 카멜표기법을 사용하여 class명과 구분을 쉽게 한다.
ex) id="contentWrap"

※카멜표기법(Camel Notation) : 카멜표기법은 단어와 단어 사이를 대소문자로 구분하는 방법으로 낙타의 혹을 닮아서 붙여진 표기방법이다.
Camel Notation uses capital letters to indicate the break between words when more than one word is used to define a variable. Camel is named such because the capital letters separating the words look like little camel humps.

3. Class명

클래스명은 하이픈(-)방식을 사용하여 연결하며 첫글자는 소문자로 시작하고 숫자, 기호 등을 사용하지 않는다.
ex) class="insurance-guide"

4. Image명

이미지명은 이미지 분류 목적에 따라 prefix_를 사용하여 구분하며 상세내용은 아래를 참고한다. 두 가지 사항이 겹칠 경우 아래의 분류를 우선해서 순차적으로 구분한다.
ex) 탭의 배경이미지 : tab_bg_000.png, 버튼안의 아이콘 : btn_icon_000.png
이미지 저장 방식은 좌측 "이미지 저장 가이드"를 참조한다.

비주얼 : v_ / 버튼 : btn_ / 배경 : bg_ / 탭 : tab_ / 아이콘 : icon_ / 메뉴 : menu_ / 블릿 : bl_ / 페이징 : pg_ / 텍스트 : txt_ / 레이어 : ly_ / 라인 : ln_ / 배너 : bnn_

5. 변수명

스크립트 작성 시 필요한 경우 사용하는 변수명은 언더스코어(_) 방식을 사용하여 id, class명과 구분한다.
ex) var section_height

접근성 확인 사항

1. 인식의 용이성(Perceivable)

1) 대체텍스트

  • 이미지의 alt 성격 구분 - 이미지 요소의 alt속성은 필수요소이며 의미있는 이미지인 경우 이미지가 제공하는 콘텐츠를 그대로 넣고, 의미가 없는 블릿이나 배경은 alt속성의 값을 비워둔다.

    ex) alt=""

  • input type="image"라면 alt 확인 - 이미지버튼인 경우 input type="image"를 사용할 때 alt 제공한다.

    ex) input type="image" src="" alt="확인"

  • 이미지맵 alt 확인 - 이미지맵 사용은 최대한 지양하되 불가피하게 사용해야할 경우 각각 area별로 alt 제공한다.

    ex)<area alt="ENGLISH" shape="rect"coords="10,5,66,19" href="/en/">

  • 이미지, 그래프, 차트에 설명문 제공 - 이미지 또는 그래프나 차트 콘텐츠에는 반드시 대체정보(row data, 대체텍스트)가 제공되어야 한다. 스토리보드에 명시되어있지 않다면 기획쪽에 문의할것
  • alt, title 구분할것 - alt와 title속성의 목적은 다르다. 이미지에 대한 대체텍스트는 alt 속성을 이용하며 부가적인 설명이 필요한 경우 title 속성을 이용한다.

    ex) img src="" alt="자세한정보보기" title="새창이동"

2) 멀티미디어대체수단

  • 동영상 콘텐츠 자막,원고 또는 수화 제공 - 홍보나 광고 소개 페이지의 경우 기존의 동영상 컨텐츠의 접근성을 확보할 수 있도록 수정이 필요하다. 현업의 확인이 필요하므로 스토리보드에 이런 컨텐츠가 있는 경우 PL에게 확인한다.

3) 명료성

  • 그래프, 차트 이미지 컨텐츠의 시각적 구분 - 그래프나 차트의 경우 색상만으로 범례구분이 된 경우 흑백 디스플레이 사용자나 흑백 인쇄물을 보는 사용자에게 문제가 되므로 디자이너에게 수정을 요청해야 한다.
  • 필수항목 구분 - 필수입력 항목의 경우 시각적 구분이 색상만으로 된 경우 흑백 디스플레이 사용자나 흑백 인쇄물을 보는 사용자에게 문제가 되므로 디자이너에게 수정을 요청해야 한다.
  • 링크 텍스트 시각적 구분 - 링크된 텍스트의 경우 시각적으로 명확히 구분될 수 있도록 underline이나 color 구분을 줘야한다. css로 없애지 않는다.
  • 오류메세지 시각적 구분 - 오류 메세지를 빨간색으로만 뿌려주는 식으로 컬러로만 구분하지 않고 오류임을 명확하게 인식하도록 제공해야 한다.(오류명시, 굵은 글꼴, 테두리 사용 등)
  • 시각에 의존한 문구 확인 - 시각에 의존한 문구가 사용된 경우 기획팀에 수정을 요청한다.

    ex) 오른쪽 확인 버튼을 누르세요, 좌측 하단 링크를 클릭하세요, 빨간테두리의 안내문구를 확인하세요

  • 의미가 있는 이미지역할을 하는 특수기호 구분 - 의미가 있는 이미지역할을 하는 특수기호 구분인 경우 HTML Entity사용보다 이미지로 제공하고 대체텍스트를 제공하는 것이 올바르다.
  • 텍스트가 작고 흐린 경우 - 텍스트의 크기가 작고 흐린 경우 디자이너에게 요청해 명도대비를 확인한다.
  • 동영상 콘텐츠의 정지기능 - 동영상콘텐츠의 경우 처음부터 실행되지 않도록 정지상태를 디폴트로 제공한다.

2. 이해의 용이성 (Understandable)

1) 가독성

  • 기본 언어 표시 - 사용하는 언어를 명시하며 PL은 HTML초기 파일에 삽입 후 배포하도록 한다.

2) 예측 가능성

  • 서식 입력 후 제출 버튼은 반드시 type="submit" 버튼 제공 - 스크립트로만 서식을 전송되도록 하지 않고 명확히 서식값 제출 버튼을 제공한다.
  • 새창 알림 - 새창으로 열리는 링크는 title로 "새창이동" 이라고 명시해준다.
  • 페이지가 로드될 때 자동으로 팝업 창이 열리는 경우 - 페이지가 로드될 때 자동으로 팝업이 열리지 않도록 한다.(운영 시 주의사항)

3) 콘텐츠의 논리성

  • 메뉴 구조 작성 시 ul li 구조 선형화 - 서브 메뉴의 경우 상위 메뉴과 논리적 관계성을 반영하여 마크업한다.
  • 레이어팝업의 구조 선형화 - 레이어팝업의 구조도 클릭 유발 컨텐츠의 바로 다음에 위치하여 논리적으로 마크업한다.
  • 빈 공백으로 글자 간격 조정 금지 - 공백특수기호를 이용하여 공백을 처리하지 않는다. (화면낭독프로그램에서 공백을 넣으면 다르게 발음될 수 있음) 이런 경우 css의 letter-spacing 속성이나 text-align 속성을 이용해 조절한다.
  • 테이블의 경우 필수 속성들을 준수 - 테이블의 경우 필수 속성( caption, scope, thead, tbody, id, headers 등)들을 준수하여 기입하며 DTD가 HTML5이니 summary속성은 사용하지 않으며 대신 caption이나 요약정보를 별도로 제공한다.
